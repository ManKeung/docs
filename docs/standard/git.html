# Git规范

提示

不管是团队还是个人开发，规范的提交代码是非常重要的

# 注意事项必要

  1. 开发之前都应该拉下代码。
  2. 每天至少保证下班提交一次代码[1]。
  3. 当有一个小功能或则bug修复就应当提交代码，不要编码了很多代码后才提交，这样不利于代码评审。

# git开发流程必要

如图所示：开发分支根据主分支建立，个人在开发分支建立分支开发，开发完成合并到开发分支，在开发分支新建合并分支，没问题再合主分支，当线上有bug在主分支建立bug修复分支，修复完成再合并到开发分支，再开发分支到合并分支再到主分支，最后删除bug修复分支。

# 代码提交强烈推荐

'dev' // 开发中
'feat' // 新特性
'improvement' // 加强现有特性
'fix' // 修补bug
'refactor' // 重构
'docs' // 文档
'test' // 单元测试
'config' // 配置文件
'style' // 格式需改
'perf' // 性能提升
'ci' // ci 主要目的是修改项目继续集成流程(例如 Travis，Jenkins，GitLab CI，Circle等)的提交
'revert' // 版本回退
'chore' // 其他修改
'scope' // commit 影响的范围（选填） 比如: route component utils build

例子

# type后面空格
git commit -m 'dev: 描述信息'

注释
[1] 这里提交一次是表示提交到代码仓库，不是本地提交。
